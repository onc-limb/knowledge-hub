## はじめに

つい考え事をする時、**無意味に画面をスクロールしていませんか？**

Twitter のタイムラインを見るでもなくただボーッとスクロール。  
Slack のチャンネルをなんとなくスクロール。  
Notion のドキュメントを読むでもなくスクロール。

そんな「無意識のスクロール癖」を持つあなた（と筆者）のために、**スクロールそのものをゲームにしてしまいました。**

その名も **「スクロールタイムアタック」** 🏃💨

ルールは簡単。**ただ 300m スクロールするだけ。**  
それだけです。本当にそれだけです。

![スクロールタイムアタックのスクリーンショット](./solid-project/public/images/scroll-trial-demo.gif)

300m スクロールしたあなたには、達成時間と特別なメッセージが贈られます。

## なぜこんなものを作ったのか

### くだらないから

クソアプリアドベントカレンダーに相応しい、**圧倒的にくだらないアプリ**を作りたかったからです。

「ゲーム」と名乗っていますが、戦略もスキルも不要。  
ただひたすらスクロールするだけ。
誰かと競うわけでもなく、自己ベストを目指すわけでもない。
あなたの貴重な時間を、何の生産性もなく消費します。

### データ非依存の技術を触りたかった

普段はバックエンド、インフラをメインに扱っている筆者。  
データベース、API、Terraform... そういった「データを中心とした世界」にいます。

しかし最近、ふと思うことがあります。

**生成AIの発展によって、世界のデータはより一層ビッグテックに握られるようになりました。**

個人開発者がGoogleやOpenAIと同じ規模のデータを持つことは不可能です。  
データによる差別化は、もはや個人には難しい時代になったのかもしれません。

だからこそ、**データに依存せず、ブラウザ上だけで完結する体験**に興味が湧きました。

特に魅力を感じたのが **Three.js**。  
3D 表現、アニメーション、視覚的フィードバック... これらは完全にフロントエンドの領域で、データの量や質に左右されません。

**「バックエンドエンジニアが、データのいらない世界で何か作るとどうなるのか？」**

そんな実験的な気持ちで、くだらないゲームを作りながら学んでみることにしました。

## 技術スタック

シンプルかつモダンな構成を心がけました。

- **SolidStart**: 軽量でリアクティブなフロントエンドフレームワーク(ただ使いたかっただけ)
- **Three.js**: WebGL ベースの 3D グラフィックスライブラリ
- **TypeScript**: 型安全な開発

データベースも API も不要。純粋にクライアントサイドだけで完結します。

## 実装のポイント

### 1. スクロール量の計測

まず最初に悩んだのが、**「どうやってスクロール量を正確に測るか？」** でした。

#### `scroll` イベントじゃダメなの？

最初は素直に `scroll` イベントを使っていました。

でも、会社の人に触ってもらったところ、**思わぬ問題が発覚**。

- ⌘+↓ で一気に最下部までスクロールされる
- スクロールバーを掴んで移動させられる
- キーボードの矢印キーでスクロールされる

これらは「ゲーム」として望ましくない操作方法でした。

今回は「**マウスホイール/トラックパッド/タッチスワイプだけ**」を計測したかったので、`wheel` イベントを採用しました。

**実際に使ってもらって初めて気づく仕様の穴。フロントエンド開発の難しさを痛感しました。**

```typescript
const handleWheel = (e: WheelEvent) => {
  e.preventDefault();

  // 下方向へのスクロールのみ計測（deltaY > 0）
  if (e.deltaY > 0) {
    accumulatedScrollPixels += e.deltaY;
  }

  // ピクセルをメートルに変換
  const meters = accumulatedScrollPixels * pixelToMeter();
  setScrollDistanceMeters(meters);
};
```

#### ピクセルをメートルに変換する

スクロール量は `deltaY` というピクセル単位で取得されます。  
でも「300 ピクセル」より「300 メートル」の方が直感的ですよね？

そこで、**ディスプレイの物理的なサイズを推定**してピクセルをメートルに変換しました。

```typescript
const pixelToMeter = () => {
  const dpi = 96 * window.devicePixelRatio; // 標準DPI × デバイスピクセル比
  const pixelsPerInch = dpi;
  const pixelsPerMeter = pixelsPerInch * 39.3701; // 1メートル = 39.3701インチ
  return 1 / pixelsPerMeter;
};
```

Retina ディスプレイなど高解像度ディスプレイでは `devicePixelRatio` が 2 以上になるため、これを考慮しないと端末によって「進んだ距離」が変わってしまいます。

#### ハマったポイント

「スクロールなんて簡単でしょ」と思っていた過去の自分を殴りたい。

- `scroll` vs `wheel` の違い
- `deltaY` の単位がブラウザ・デバイスで異なる
- スクロールバーのドラッグをどう扱うか

バックエンドなら入力値は API のリクエストボディとして決まった形式で来ますが、フロントエンドは **ユーザーがブラウザで何をするか予測するのが難しい**。

実際に使ってもらうことで初めて気づく仕様の検討漏れが多く、「想定外の操作」への対応が必要でした。

### 2. Three.js で視覚的フィードバック

「ただスクロールするだけ」だと味気ないので、**Three.js で幾何学的なパターンが流れる背景**を作りました。

背景は

- 黄金角スパイラル状に配置した大量のパーティクル
- 奥行き方向に並んだ複数のリング

の 2 レイヤー構成です。パーティクルは `ShaderMaterial` を使い、頂点シェーダ内で「時間」「スクロール量」「スクロール速度」に応じて位置を揺らしています。

スクロール入力は `updateByScroll` で「距離」と「速度」に変換し、`tick` でなめらかに補間してから、シェーダやリング、カメラの揺れに一括で反映しています。

```typescript
updateByScroll(scrollMeters: number) {
  const delta = scrollMeters * 0.15;
  this.targetScrollY = delta;
  const rawVelocity = (delta - this.scrollY) * 8;
  this.scrollVelocity = Math.max(-2, Math.min(2, rawVelocity));
}
```

「速くスクロールすると背景が激しく動き、止めると徐々に静まる」という挙動になっていて、**スクロールそのものにだけ反応する 3D 背景**になっています。

## 所感: 学習のための「無駄」は無駄ではない

### Coding Agent に任せたら一瞬なのに

正直、このゲーム程度なら ClaudeCode や CodeX などの Coding Agent に頼めば数分で完成するでしょう。

でも、**自分の手で作るという労力が人を強くする**のだと思います。

- スクロールイベントの違いに悩む
- Three.js の座標系で混乱する
- デバイスごとの挙動の違いに気づく

こういった「つまずき」と「解決」の積み重ねが、本当の理解につながります。

### 使ってもらうことで気づく仕様の検討漏れ

バックエンド開発では、API の入力は決まった形式で来ます。  
でも、**ユーザーが手元のブラウザで何をするか想定するのは、バックエンドよりも遥かに難しい**。

- スクロールバーをドラッグする人
- トラックパッドで微妙に逆スクロールする人
- 画面を拡大してプレイする人

実際に使ってもらって初めて、「この操作、想定してなかった！」と気づくことが多々ありました。

### 学習のための無駄は、無駄ではない

このゲームは何の役にも立ちません。  
プレイしても生産性は上がらないし、スキルも身につかない。

**でも、作るプロセスは確実に自分を成長させてくれました。**

現代は「生産性」「効率化」「AI に任せる」ばかり。  
でも、たまには非効率でも自分の手を動かして、つまずいて、学ぶ。

**そういう「回り道」が、実は一番の近道だったりするんじゃないか。**

## おわりに

**「無意味にスクロールするあなたへ贈る、スクロールするだけのゲーム」**

くだらないものを作りながら、Three.js やフロントエンドの世界を学ぶことができました。

Coding Agent に任せれば一瞬で終わる作業を、わざわざ自分の手で作る。  
非効率で、遠回りで、時には挫折しそうになる。

**でも、その過程こそが最高の学びでした。**

あなたも、考え事をしながら無意味にスクロールする時は、  
ぜひこのゲームで「公式に無駄にして」ください。

**あなたの貴重な時間を無駄にしてくれて、ありがとうございます。** 🙏
